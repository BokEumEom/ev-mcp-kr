# Architecture Decision Records (ADR)

ev-mcp 의 굵직한 아키텍처 결정을 박제하는 곳. 결정·근거·대안·롤백 계획을 한 파일에.

## 언제 ADR 을 쓰는가

다음 중 하나에 해당하면 ADR 신설:

- 새 외부 시스템 도입 (DB, 호스팅, 큐, 캐시 등)
- 기존 시스템 폐기·교체
- 데이터 모델의 진실의 원천 변경
- 운영 비용 또는 SLA 가 바뀌는 결정
- 보안 모델 변경 (인증/인가/시크릿)
- 한 사람이 "왜 이렇게 했지?" 라고 6개월 후 물을 만한 결정

작은 라이브러리 추가, 함수 시그니처 변경 같은 건 PHASE 보고서로 충분.

## 파일명 규칙

`ADR-{NNN}-{kebab-case-제목}.md` — NNN 은 3자리 zero-pad. 번호는 시간 순서대로 한 번 부여 후 변경하지 않는다 (재배치 X).

## 상태

| 상태 | 의미 |
|---|---|
| **Proposed** | 초안. 아직 적용 X. |
| **Accepted** | 결정 확정. 구현 중이거나 운영 중. |
| **Superseded by ADR-NNN** | 후속 결정에 의해 대체됨. 본문은 보존 (역사). |
| **Deprecated** | 결정 자체가 무효화. 보존 (왜 무효화됐는지 명시). |

## 작성 템플릿

```markdown
# ADR-NNN — 한 줄 결정 요약

**상태:** Proposed | Accepted (YYYY-MM-DD)
**결정자:** {이름}
**대상 Phase:** Phase N (관련 있다면)
**관련 문서:** {연결 ADR, PoC, PHASE 보고서}

## 맥락
왜 이 결정이 필요한가. 현재 상태와 문제.

## 결정
무엇을 결정했는가. 명시적으로 채택하지 않는 것도 포함.

## (해당 시) PoC 검증 결과
수치 + 표.

## 대안
검토한 다른 옵션 + 거부 이유 (각각 한 단락).

## 결과 (트레이드오프)
얻는 것 / 감수하는 것 / 운영 비용.

## 트리거 / 재검토 조건
어떤 조건이 발생하면 이 결정을 다시 본다.

## 롤백 계획
이 결정이 잘못된 것으로 판명되면 어떻게 되돌리는가. 단계별, 데이터 영향, 최종 상태.
```
